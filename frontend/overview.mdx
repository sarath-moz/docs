## What's in the Monorepo

The frontend is a **pnpm workspace monorepo** with two deployable apps and two shared packages.

  <Card title="apps/full-suite" icon="browser" href="/frontend/full-suite">

  </Card>
  <Card title="apps/a11y" icon="universal-access" href="/frontend/a11y">

  </Card>
  <Card title="packages/common" icon="boxes-stacked" href="/frontend/packages">

**Root-level files:**

| File | Purpose |
|------|---------|
| `package.json` | Workspace root — scripts, dev tooling, engines |
| `pnpm-workspace.yaml` | Declares `apps/*` and `packages/*` as workspaces |
| `.husky/` | Git hooks — runs lint-staged on every commit |
| `commitlint.config.mjs` | Enforces Conventional Commits |
| `.prettierrc.json` | Prettier config (double quotes by default) |
| `eslint.config.js` | Flat ESLint config for the whole repo |

## Package Manager: pnpm Workspaces

**pnpm v9.15.2** — Node ≥ 20.14.0 required.

The root `package.json` exposes shorthand scripts to run commands scoped to a specific workspace:

| Script | Equivalent to |
|--------|--------------|
| `pnpm suite <cmd>` | `pnpm --filter full-suite <cmd>` |
| `pnpm a11y <cmd>` | `pnpm --filter a11y <cmd>` |
| `pnpm common <cmd>` | `pnpm --filter common <cmd>` |
| `pnpm focus-order <cmd>` | `pnpm --filter @webyes/focus-order <cmd>` |

```bash
pnpm suite dev        # Start full-suite dev server
pnpm a11y build       # Build the a11y app
pnpm common build     # Build the common package
pnpm clean            # Remove all node_modules
```

  </Accordion>

  <Accordion title="Husky + lint-staged (pre-commit)">
    Every `git commit` automatically triggers:

    1. `eslint --fix` on all staged `**/*.{js,ts,tsx}` files
    2. `prettier --write` on all other staged files

    The hook is defined in `.husky/pre-commit` and configured via `lint-staged` in the root `package.json`.
  </Accordion>

  <Accordion title="Commitlint (Conventional Commits)">
    Every commit message must follow the format:

    ```
    <type>(<optional scope>): <subject>
    ```

    **Valid types:** `feat` · `fix` · `docs` · `style` · `refactor` · `test` · `chore` · `perf`

    ```bash
    git commit -m "feat: add scan history pagination"   # ✅
    git commit -m "fixed stuff"                          # ❌ rejected
    ```
  </Accordion>

## How Packages Are Shared

Apps reference shared packages via the **pnpm workspace protocol** — no build or publish step needed during development:

```json
{
  "dependencies": {
    "common": "workspace:*",
    "@webyes/focus-order": "workspace:*"
  }
}
```

pnpm symlinks these automatically. Changes in `packages/common` are immediately reflected in both apps.

## Build Pipeline

Each app is built with **Vite 7 + SWC**:

```bash
pnpm suite build    # TypeScript check → Vite build → apps/full-suite/dist/
pnpm a11y build     # TypeScript check → Vite build → apps/a11y/dist/
```

The `@tanstack/router-vite-plugin` runs automatically during every build and dev session:

<Steps>
  <Step title="Scan routes folder">
    Reads all files inside `src/routes/` and derives the URL structure from the file names.
  </Step>
  <Step title="Generate route tree">
    Writes `src/routeTree.gen.ts` — **never edit this file manually**, it is always overwritten.
  </Step>
  <Step title="Enable code splitting">
    Routes ending in `.lazy.tsx` are split into separate JS chunks, loaded only when visited.
  </Step>
</Steps>

## Environment Variables

Each app has its own `.env` (gitignored) and `.env.sample` (committed).

<Warning>
  All variables **must be prefixed with `VITE_`** to be accessible in the browser. Unprefixed variables are `undefined` at runtime.
</Warning>

```typescript
const API_URL = import.meta.env.VITE_API_URL   // ✅ works
const SECRET  = import.meta.env.SECRET_KEY     // ❌ always undefined
```
