## Purpose

The **a11y** app is a focused, standalone version of the auditing platform that targets **accessibility-only** use cases. It's sold as a separate product at `accessibility.webyes.com`.

**Key differences from full-suite:**

| Feature | full-suite | a11y |
|---------|-----------|------|
| Accessibility auditing | ✅ | ✅ |
| Performance metrics | ✅ | ❌ |
| SEO analysis | ✅ | ❌ |
| Uptime monitoring | ✅ | ❌ |
| RUM (Real User Monitoring) | ✅ | ❌ |
| Team management | ✅ | Limited |
| Billing / subscriptions | ✅ | Separate plans |
| Simplified onboarding | ❌ | ✅ |
| Website-level settings | ✅ | ✅ (simplified) |

**Dev URL:** http://localhost:5174
**Staging:** https://accessibility.staging.webyes.com
**Production:** https://accessibility.webyes.com

## Code Architecture

The a11y app is **nearly identical in structure** to full-suite. Same:
- Entry point pattern (`main.tsx` → `App.tsx` → TanStack Router)
- State management (Zustand + TanStack Query)
- API layer (same backend, different `VITE_API_URL` paths)
- `packages/common` for shared components
- `packages/focus-order` for accessibility tooling

The differences are only in:
1. **Route set** — fewer routes, accessibility-specific flows
2. **Environment variables** — different URLs
3. **Analytics IDs** — separate Mixpanel/PostHog project

## When to Work in a11y vs full-suite

| Task | Work in |
|------|---------|
| Fix accessibility audit display bug | Both (shared component in `common`) |
| Add a new a11y-only onboarding step | `apps/a11y` only |
| Add a performance chart | `apps/full-suite` only |
| Update the login page | `packages/common/src/components/auth/` |
| Add a shared UI component | `packages/common` |
| Update a11y billing flow | `apps/a11y/src/routes/.../billing.tsx` |
| Update suite billing flow | `apps/full-suite/src/routes/.../billing.tsx` |
